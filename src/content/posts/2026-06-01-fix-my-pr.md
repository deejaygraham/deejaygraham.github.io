Or "How to Update a Feature Branch From master and Still End Up With a Single Commit PR".
If your repository expects exactly one commit in each pull request, you need to be a bit careful 
when syncing your feature branch with the latest main branch.

I constantly forget how to do this, so as usual, it's now a blog post as a practical reminder 
for future-me: how to take an existing branch and bring it up to date with main, continue 
working, and then make sure the PR still shows as a single commit.

## The Goal

You want to:

* Update your feature branch with the latest commits from main
* Make your own code changes
* Open a PR that contains only one commit

This means avoiding a PR history that looks like this:

```bash
* fix import bug
* update mapping
* rework migration logic
* merge main into feature branch
```

Ideally we want want the PR to show just:

```bash
Add cats per square ironing board
```

## Solution

The cleanest way to do this is:

* Rebase your branch onto main
* Keep working as normal
* Squash everything into one commit before creating the PR

This avoids merge commits and keeps history tidy.


## Code

Check for uncommitted changes first 

```bash
git status
```

Get the latest from main

```bash
git fetch origin
```

Switch to the feature branch

```bash
git checkout my-wonderful-new-branch
```

Replay your feature branch commits on top of the latest main. Git takes the commits that exist on your 
feature branch, temporarily removes them, moves your branch pointer to origin/main, and then reapplies 
your commits one by one on top. This gives you a clean, linear history.

```bash
git rebase origin/master
```

Squash down to a single commit

```bash
git reset --soft origin/main
git commit -m "Added some awesome stuff"
```

Push 

```bash
git push --force-with-lease origin my-wonderful-new-branch
```
